Core Concept Weeks 1–2

Software Development Life Cycle

The SDLC is the structured process that guides how software is planned, created, tested and maintained. Understanding it is the foundation of professional software engineering.

What is SDLC?

📌 Definition

The Software Development Life Cycle (SDLC) is a systematic process for planning, creating, testing, and deploying software. It provides a structured framework that breaks development into distinct phases, each with clear goals, deliverables, and responsibilities.

Why it matters: Without a structured lifecycle, projects tend to go over budget, miss deadlines, and ship buggy software. The SDLC exists to bring predictability and quality to a complex creative process.

The 7 Phases

  1. 1

    Planning

    Define project scope, timeline, budget, and team. Identify feasibility and risks. Output: Project Plan, Feasibility Study.

  2. 2

    Requirements Analysis

    Gather and document what the system must do (functional) and how it should perform (non-functional). Output: Software Requirements Specification (SRS).

  3. 3

    System Design

    Translate requirements into architecture. Define database schema, UI wireframes, system components. Output: Design Document, UML diagrams.

  4. 4

    Implementation (Coding)

    Developers write code based on the design. This is where Git, code reviews and Agile sprints happen. Output: Working code + tests.

  5. 5

    Testing

    Verify the software does what the requirements say. Unit tests, integration tests, UAT (user acceptance testing). Output: Test reports, bug fixes.

  6. 6

    Deployment

    Release software to production. Could be a phased rollout, blue-green deployment or full release. Output: Live running system.

  7. 7

    Maintenance

    Fix bugs, add features, adapt to new requirements over the software's lifetime. This phase is often the longest and most expensive.

Types of SDLC Models

🌊 Waterfall Model

Phases flow sequentially. Each phase must complete before the next begins. Good for projects with fixed, well-understood requirements. Bad when requirements change mid-project.

🔁 Iterative / Agile

Development happens in short cycles (sprints). Requirements evolve. Feedback is gathered after each iteration. Most modern software teams use this. Covered in depth in Process Models.

🌀 Spiral Model

Combines iterative development with risk analysis. Each spiral covers planning → risk analysis → development → evaluation. Best for large, high-risk projects.

Waterfall vs Agile — Quick Comparison

Aspect Waterfall Agile
Requirements Fixed upfront Evolve throughout
Delivery One big release at end Small releases every sprint
Customer involvement Low (beginning + end) High (every sprint)
Change handling Expensive and difficult Expected and embraced
Best for Construction, compliance software Web apps, startups, products
Documentation Heavy upfront docs Just enough documentation

Exam tip: When asked to choose a model in an exam scenario — look for keywords. "Fixed requirements + government project" → Waterfall. "Unclear requirements + startup + user feedback" → Agile.

Key Terms to Know

📄 SRS — Software Requirements Specification Document

The formal document that captures all functional and non-functional requirements. It's the contract between developers and stakeholders about what the software will do.

✅ Definition of Done (DoD)

A shared checklist that defines when a feature is truly complete. Example: code written + reviewed + unit tested + deployed to staging = Done. Agile teams live by this.

🔄 CI/CD — Continuous Integration / Continuous Deployment

A practice where code changes are automatically tested and deployed. Every commit triggers tests; if they pass, the code ships. Reduces "it works on my machine" problems.
