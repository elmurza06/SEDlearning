Version Control, Code Quality & Agile

Professional engineers live by three skills: disciplined Git workflow, readable code that teams can maintain, and Agile execution that ships working software incrementally.

Key formula from the course: Professional Engineer = Git Discipline + Readable Code + Agile Execution

What is Version Control?

🌿 Git as a Commit Graph

Git tracks history as a directed acyclic graph (DAG) — each node is a snapshot of your code at a point in time. Every commit points to its parent, letting you navigate your entire project history, revert changes, and collaborate without overwriting each other's work.

The Branching Model

🔒 main — always stable

The main branch should always contain production-ready, tested, reviewed code. Never commit directly to main. This is a hard rule in professional teams.

✨ feature/* branches

Create a new branch for every feature. Example: git checkout -b feature/user-auth. When done, open a Pull Request to merge it back. Keeps work isolated and reviewable.

🐛 fix/* branches

Use separate branches for bug fixes. Keeps your work organized and the history readable. Example: git checkout -b fix/login-null-crash.

Working Tree vs Staging

# Working tree: you edited the file git status # shows modified files # Staging: mark what you want to commit git add src/auth/login.dart # Committing: save a snapshot git commit -m "feat(auth): add email validation" # Rule: only commit what you fully understand

Good vs Bad Commits

Bad ❌Good ✅
"fix""fix(api): handle null response from user endpoint"
"update""feat(auth): add email validation with regex"
"changes""refactor(db): extract user query into repository"
"wip""test(login): add edge cases for empty password"

Format: type(scope): description — Types: feat, fix, refactor, test, docs, chore. This is called Conventional Commits and is used at Google, Microsoft, and most professional teams.

Pull Request Flow

  1. 1

    Push feature branch

    git push origin feature/my-feature — uploads your local work to remote so others can see it.

  2. 2

    Open Pull Request (PR)

    On GitHub/GitLab, create a PR to merge your branch into main. Write a clear description of what changed and why.

  3. 3

    Code Review

    Teammates review for readability, logic, edge cases, naming. Comments should be constructive: "Can we extract this into a method?" not "This is bad."

  4. 4

    Fix feedback

    Address reviewer comments. Push fixes to the same branch — the PR updates automatically.

  5. 5

    Merge

    Once approved, merge into main. Delete the feature branch to keep the repo clean.

Code Quality — The Google Mindset

"Code is read much more than it's written." — The goal is to write code that your future self and teammates can understand without guessing.

1. Clear Names

// Bad int x(int a) => a * 2; // Good int calculateDoubleScore(int score) => score * 2;

2. Small Functions — One Job

// Bad: one function does 3 things void processUser(User u) { validate(u); saveToDb(u); sendEmail(u); } // Good: clear names reveal intent void registerUser(User user) { validateUser(user); saveUser(user); sendWelcomeEmail(user); }

3. No Magic Numbers

// Bad: what does 18 mean? if (age > 18) { ... } // Good: self-documenting const legalAge = 18; if (age > legalAge) { ... }

4. Consistency

// Bad: mixed naming styles get_user_name() getUserAge() USER_EMAIL() // Good: one consistent style getUserName() getUserAge() getUserEmail()

Agile Engineering

Agile ≠ meetings. Agile is not about ceremonies or standups. It's about delivering working software in small increments, gathering feedback quickly, and adapting.

📋 Sprint Board: ToDo → In Progress → Review → Done

Every task moves through columns. Team visibility ensures accountability and surfaces blockers early. Tools: GitHub Projects, Jira, Linear, Notion.

✅ Definition of Done (DoD)

A feature is Done when:
Code Written — follows style guidelines, no magic numbers, clear naming.
Reviewed — peer-reviewed, all comments addressed, approved.
Tested — unit tests passing, edge cases covered, no regressions.

🧍 Daily Standup Format

What did I do? — completed tasks, merged PRs, resolved issues.
What will I do? — planned tasks for today.
Blockers? — dependencies, unclear requirements, technical issues.

Merge vs Rebase

git mergegit rebase
Preserves full history, adds a merge commitRewrites history as if changes were made on top of main
History is more honest but can look messyCreates a cleaner, linear history
Safe for shared branchesNever rebase shared/public branches
Use for: merging feature into mainUse for: cleaning up local commits before PR
