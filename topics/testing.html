Software Testing Fundamentals

Testing is not about finding bugs — it's about confidence. A good test suite lets you change code fearlessly, knowing it will tell you when something breaks.

Key insight: Testing is a first-class engineering activity, not an afterthought. Professional teams write tests as they write code — or even before (TDD).

Types of Testing

🔬 Unit Testing

Tests the smallest piece of code in isolation — a single function or method. Fast to run, easy to pinpoint failures. Should cover the happy path, edge cases, and error cases. Frameworks: JUnit (Java), pytest (Python), flutter_test (Dart).

🔗 Integration Testing

Tests how multiple components work together — a database + service + API. Slower than unit tests but catches the "it works separately but not together" class of bugs.

🖥️ System Testing

Tests the entire application as a user would. Also called end-to-end (E2E) testing. Slowest and most expensive, but validates real user flows. Tools: Selenium, Cypress, Playwright.

👥 User Acceptance Testing (UAT)

Done by actual users or clients. Validates that the system meets business requirements. Last gate before production release.

Testing Pyramid

🔺 The Testing Pyramid

The most famous model in software testing:

  • E2E TestsFew — slow, expensive, but test real user flows
  • IntegrationSome — test component interactions
  • Unit TestsMany — fast, cheap, precise; the foundation

The more tests you have at the bottom, the faster and more reliable your test suite. Inverting the pyramid (too many E2E, few unit) = slow, brittle CI pipeline.

Test-Driven Development (TDD)

TDD cycle: Red → Green → Refactor. Write a failing test first, then write the minimum code to make it pass, then clean up the code.

  1. R

    Red — Write a failing test

    Write a test for functionality that doesn't exist yet. It must fail. This clarifies exactly what you're building.

  2. G

    Green — Make it pass

    Write the minimum possible code to make the test pass. Don't write more than needed — YAGNI (You Ain't Gonna Need It).

  3. R

    Refactor — Clean up

    Improve the code without changing behavior. The tests protect you — if they still pass after refactoring, you haven't broken anything.

// TDD example in Dart / Flutter // Step 1: RED — write failing test test('calculateTotal adds items correctly', () { expect(calculateTotal([10.0, 20.0, 5.0]), equals(35.0)); }); // Step 2: GREEN — implement function double calculateTotal(List<double> prices) { return prices.fold(0.0, (sum, p) => sum + p); } // Step 3: REFACTOR — this is already clean ✓

Key Testing Concepts

📊 Code Coverage

The percentage of code lines/branches executed by tests. 80% is often cited as a target but it's not a magic number — 80% coverage of the wrong things is worse than 60% of the critical paths. Coverage is a proxy for quality, not quality itself.

🎭 Mocking & Stubs

Replacing a real dependency (like a database or API) with a fake one for testing. A stub returns preset values; a mock can also verify the right calls were made. Essential for unit testing without side effects.

🔧 Regression Testing

Running existing tests after changes to make sure nothing broke. CI/CD pipelines automate this — every commit triggers the full test suite automatically.

Exam tip: Know the difference between Verification ("are we building the product right?") and Validation ("are we building the right product?"). Verification = internal quality checks. Validation = matches user needs.
